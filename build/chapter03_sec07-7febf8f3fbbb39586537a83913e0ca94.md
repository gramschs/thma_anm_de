# 3.7 Rechenzeit und Skalierung linearer Gleichungssysteme

Wir haben `np.linalg.solve()` bisher für kleine Systeme mit 3 oder 6
Unbekannten verwendet. In der Ingenieurpraxis — etwa bei der
Finite-Elemente-Simulation eines Motorblocks oder der CFD-Analyse einer
Turbinenschaufel — entstehen Systeme mit Tausenden oder Millionen von
Unbekannten. Wie lange dauert es, solche Systeme zu lösen? Und wie
wächst die Rechenzeit mit der Systemgröße?

## Zufällige, eindeutig lösbare Gleichungssysteme erzeugen

Für unsere Zeitmessung brauchen wir Testprobleme: Matrizen $\mathbf{A}$
und Vektoren $\vec{b}$ beliebiger Größe, die ein eindeutig lösbares
System bilden. Eine zufällige Matrix ist fast immer invertierbar (die
Wahrscheinlichkeit, zufällig eine singuläre Matrix zu erzeugen, ist
praktisch null). Wir sichern das dennoch ab:

```{code-cell} python
import numpy as np
import time
import matplotlib.pyplot as plt
import matplotlib.style as style
style.use('seaborn-v0_8')

def erzeuge_lgs(n, seed=None):
    """Erzeugt ein zufälliges, eindeutig lösbares LGS der Größe n×n.

    Parameter
    ----------
    n : int
        Dimension des Gleichungssystems
    seed : int oder None
        Zufallsseed für Reproduzierbarkeit

    Rückgabe
    --------
    A : ndarray, Shape (n, n)
        Koeffizientenmatrix
    b : ndarray, Shape (n,)
        Rechte Seite
    """
    rng = np.random.default_rng(seed)

    # Zufällige Matrix
    A = rng.standard_normal((n, n))

    # Eindeutigkeit sicherstellen: Diagonaldominanz durch Addition von n*I
    # Eine diagonaldominante Matrix ist immer invertierbar.
    A += n * np.eye(n)

    # Zufälliger rechter Seiten-Vektor
    b = rng.standard_normal(n)

    return A, b


# Test für n = 5
A_test, b_test = erzeuge_lgs(5, seed=42)
print(f"Form von A: {A_test.shape}")
print(f"Determinante: {np.linalg.det(A_test):.2f}")
print(f"Rang: {np.linalg.matrix_rank(A_test)}")
```

```{admonition} Diagonaldominanz
:class: note
Eine Matrix heißt **diagonaldominant**, wenn jedes Diagonalelement
dem Betrag nach größer ist als die Summe aller anderen Elemente in
derselben Zeile. Diagonaldominante Matrizen sind immer invertierbar.
Durch Addition von $n \cdot \mathbf{I}$ fügen wir $n$ zu jedem
Diagonalelement hinzu, was bei standardnormalverteilten Einträgen
(typische Werte um 0, selten größer als 3) Diagonaldominanz
sicherstellt.
```

## Rechenzeit messen

Wir schreiben eine Funktion, die ein LGS der Größe $n$ erzeugt, löst
und die dafür benötigte Zeit zurückgibt:

```{code-cell} python
def messe_rechenzeit(n):
    """Erzeugt ein zufälliges n×n-LGS, löst es und gibt die
    Rechenzeit in Sekunden zurück.

    Parameter
    ----------
    n : int
        Dimension des Gleichungssystems

    Rückgabe
    --------
    t_elapsed : float
        Rechenzeit in Sekunden
    """
    A, b = erzeuge_lgs(n)

    t_start   = time.time()
    x         = np.linalg.solve(A, b)
    t_elapsed = time.time() - t_start

    return t_elapsed


# Einzelmessung für n = 500
t = messe_rechenzeit(500)
print(f"Rechenzeit für n = 500: {t*1000:.2f} ms")
```

## Parameterstudie: Rechenzeit vs. Systemgröße

Wir messen die Rechenzeit für ausgewählte Systemgrößen von $n = 100$
bis $n = 3000$:

```{code-cell} python
# Ausgewählte Systemgrößen
n_werte = np.array([100, 250, 500, 750, 1000, 1500, 2000, 3000])
t_werte = np.zeros(len(n_werte))

print("Messe Rechenzeiten ...")
for i, n in enumerate(n_werte):
    t_werte[i] = messe_rechenzeit(n)
    print(f"  n = {n:5d}: {t_werte[i]*1000:.1f} ms")

print("Fertig.")
```

```{admonition} Hinweis zur Rechenzeit
:class: note
Die gemessenen Zeiten hängen stark von der Hardware ab und können bei
jedem Durchlauf leicht variieren, da das Betriebssystem andere Prozesse
parallel ausführt. Für eine sicherere Messung würde man jeden Punkt
mehrfach messen und den Mittelwert nehmen. Für unsere qualitative
Analyse der Skalierung reicht eine Einzelmessung.
```

## Visualisierung

```{code-cell} python
fig, ax = plt.subplots(figsize=(8, 5))

# Gemessene Rechenzeiten (log-log)
ax.loglog(n_werte, t_werte, color='#4C72B0', linewidth=2,
          marker='o', markersize=7, label='gemessen')

# Theoretische O(n³)-Referenz: skaliert auf den ersten Messpunkt
t_ref = t_werte[0] * (n_werte / n_werte[0])**3
ax.loglog(n_werte, t_ref, color='#DD8452', linewidth=2,
          linestyle='dashed', label='$O(n^3)$-Referenz')

ax.set_xlabel('Systemgröße n')
ax.set_ylabel('Rechenzeit in s')
ax.set_title('Rechenzeit von np.linalg.solve (log-log-Skala)')
ax.legend()
ax.grid(True, which='both')

plt.tight_layout()
plt.show()
```

### Interpretation

In der log-log-Darstellung entspricht eine gerade Linie einem
Potenzgesetz $t \propto n^\alpha$. Die Steigung der Kurve gibt den
Exponenten $\alpha$ an — eine Steigung von 3 entspricht $O(n^3)$.

```{code-cell} python
# Numerische Abschätzung der Steigung im log-log-Raum
log_n = np.log(n_werte)
log_t = np.log(t_werte)

# Lineare Regression im log-log-Raum (nur obere Hälfte der Daten)
mitte    = len(n_werte) // 2
steigung = np.polyfit(log_n[mitte:], log_t[mitte:], 1)[0]
print(f"Geschätzter Skalierungsexponent: {steigung:.2f}")
print(f"Theoretischer Wert (LU-Zerlegung): 3.00")
```

Der gemessene Exponent liegt nahe bei 3, was die theoretische
$O(n^3)$-Komplexität der LU-Zerlegung bestätigt. Verdoppelt man die
Systemgröße, steigt die Rechenzeit etwa um den Faktor $2^3 = 8$.

````{admonition} Lösung
Ausgabe (typisch):
```
Gemessene Rechenzeit für n = 3000: 0.85 s
Geschätzte Rechenzeit für n = 10000: 31 s (0.5 min)
```

Für $n = 10^6$ ergibt die Abschätzung eine astronomisch lange Rechenzeit —
`np.linalg.solve()` ist dafür völlig ungeeignet.

In der Praxis nutzt man für große, dünn besetzte Systeme (**sparse
matrices**) iterative Löser wie den CG-Algorithmus (Conjugate Gradient)
oder GMRES. Diese sind auf die spezielle Struktur großer Ingenieurprobleme
zugeschnitten und benötigen nur $O(n)$ bis $O(n \log n)$ Operationen —
statt $O(n^3)$. In Python stehen sie über `scipy.sparse.linalg` zur
Verfügung.
````

## Zusammenfassung

Die Rechenzeit von `np.linalg.solve()` wächst mit der dritten Potenz
der Systemgröße: $O(n^3)$. Das folgt direkt aus der LU-Zerlegung, die
intern verwendet wird. Für kleine und mittlere Systeme (bis einige
tausend Unbekannte) ist das kein Problem. Für die großen Systeme der
Ingenieurpraxis — Finite-Elemente-Methode, Strömungssimulation,
Strukturmechanik — braucht man spezialisierte iterative Löser, die die
Dünnbesetztheit der Matrizen ausnutzen.
